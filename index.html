Привет, Кот. Мои ответы на теоретические вопросы за 29 неделю.

  1. Можно ли создать несколько контекстов в одном приложении?

    Ответ: Контекстов в приложении может быть несколько. В таком случае, нужно будет создать несколько независимых компонентов.

  2. Можем ли мы влиять на контекст из вложенных компонентов?

    Ответ: React Context API по умолчанию не имеет состояния и не предоставляет специальный метод для обновления значения контекста из компонентов-потребителей. Но это можно легко реализовать, интегрировав механизм управления состоянием (например, хуки useState () или useReducer ()) и предоставив функцию обновления прямо в контексте рядом с самим значением. Так вложенные компоненты смогут менять значение контекста.

  3. Как выбрать компонент, в котором будет определяться контекст?

    Ответ: Выбор компонента, в котором будет определяться контекст, зависит от конкретных требований и контекста проекта. Определите, какой именно контекст вам необходимо передавать и какие данные должны быть доступны внутри компонента. Например, если вам нужно передавать информацию о текущем пользователе, то компонент контекста может содержать данные о пользователе, а все компоненты, которым эта информация необходима, могут подписаться на контекст и получать доступ к этим данным. Обратите внимание на область видимости контекста. Некоторые компоненты могут требовать доступа к контексту только в своем непосредственном дочернем компоненте, в то время как другие могут требовать доступа к контексту в нескольких вложенных компонентах. В зависимости от этого выберите наиболее удобный и подходящий способ передачи контекста. В конечном итоге, выбор компонента контекста зависит от ваших специфических потребностей и предпочтений.

  4. Возможно ли создать контекст, который доступен только в части приложения?

    Ответ: да, в React есть несколько способов создания контекста, который будет доступен только в определенных частях приложения.

    1. Один из способов - это использование компоненты `Context.Provider` внутри корневого компонента приложения. Вы можете обернуть часть приложения, в которой хотите использовать этот контекст, внутри `Context.Provider` и передать ему значение контекста. В этом случае, контекст будет доступен только внутри обернутой части приложения.
    2. Второй способ - это использование хука `useContext` внутри компонента. Вы можете использовать этот хук в любой части приложения, чтобы получить доступ к значению контекста, переданному через `Context.Provider`.

  5. На каком этапе жизненного цикла компонента лучше всего запросить данные с сервера?

    Ответ: на этапе монтирования компонента: в методе componentDidMount в классовых компонентах или в хуке useEffect(() =>{ fetch('/api/words') .then((response) => response.json()) .then((response) => setWords(words = response)) }, []); в функциональных компонентах.

  6. Вернёт ли сообщение об ошибке Fetch API, если код ответа 401 (ошибка авторизации)?

    Ответ: нет, так как код 401 он считает валидным.

  7. Каким ещё способом можно делать запросы к API кроме fetch?

    Ответ: XMLHttpRequest (устаревающий метод), Axios, GraphQL.

  8. Зачем нужен CORS?

    Ответ: CORS — важным механизмом безопасности для веб-разработчиков, который позволяет контролировать доступ к серверным ресурсам. CORS, или Cross-Origin Resource Sharing, является механизмом, который позволяет многим ресурсам (например, шрифтам, изображениям, JavaScript) на веб-странице запрашивать данные с другого домена, отличного от домена исходной страницы. В целях безопасности, браузеры запрещают запросы к другим доменам, используя технику «одного источника» (same-origin policy). Это означает, что веб-страница, загруженная с одного домена, не может получить данные с другого домена без явного разрешения сервера. CORS позволяет серверам указать, с каких доменов разрешено получать данные, обеспечивая безопасность и контроль доступа.
